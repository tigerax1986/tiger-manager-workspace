### Completed Action
Split all multi-table migration files into one file per table. Old files deleted, new files committed and run successfully against live DB.

### New Naming Convention
- Files named `<table>.001.sql` — no numeric prefixes, version suffix for future changes
- Old: `001_create_events.sql`, `004_stadium_maintenance.sql` (multi-table monsters)
- New: `events.001.sql`, `tickets.001.sql`, `stadiums.001.sql`, `seats.001.sql`, etc.
- Future table changes → `<table>.002.sql`, `<table>.003.sql`, etc.

### Key Design Decisions
1. **No FK REFERENCES in CREATE TABLE** — tables created without foreign key deps, so alphabetical sort order doesn't matter
2. **All FKs in `zzz_constraints.001.sql`** — runs last (z sorts after all other letters), adds every FK with `IF NOT EXISTS` guard
3. **Self-contained files** — each file includes its own `CREATE OR REPLACE FUNCTION update_updated_at_column()` for triggers (idempotent)
4. **Seed data stays with its table** — stadiums seed in `stadiums.001.sql`, seat_colours defaults in `seat_colours.001.sql`, etc.
5. **stands_areas is migration bridge** — creates legacy table with seed data, then INSERT INTO stands/areas FROM stands_areas so both new and legacy tables are populated
