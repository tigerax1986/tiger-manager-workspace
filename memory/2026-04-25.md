- **Bug**: Clicking an area in view mode showed only 3 visible rows with lots of whitespace, but toggling the edit button (which triggers `watch(editMode)`) showed all rows perfectly
- **Initial fix**: Replaced `setTimeout(() => autoFitSeats(), 50)` with `nextTick(() => autoFitSeats())` in both `selectArea()` and `generateSeatsFromRowDefinitions()` — didn't fully solve it
- **Root cause**: The view-mode grid is padded to 100×100 in the DOM (via gridRows computed), but `autoFitSeats()` was being called from `selectArea()` with an explicit `nextTick` that could race against the reactive seat data update
- **Final fix**: 
  - Removed `autoFitSeats()` call from `selectArea()` entirely
  - Removed redundant `await fetchSeats()` from `selectArea()` (the `watch(selectedAreaCode)` already handles it)
  - Added a `watch(seats)` that calls `autoFitSeats()` when in view mode — uses the same reactive mechanism as `watch(editMode)` which already works correctly
  - Moved empty-area check (`seats.length === 0 → editMode = false`) into the `selectedAreaCode` watcher instead of `selectArea()`

## Auto-Fit Fix Details (watch(seats) approach)
- **Problem**: View mode grid is padded to 100×100 in the DOM. Calling `autoFitSeats()` from `selectArea()` after `nextTick` sometimes measured the canvas before the padded grid was fully laid out, or after a stale DOM state.
- **Solution**: Use Vue's reactive system — `watch(seats)` fires naturally when `fetchSeats()` completes and updates `seats.value`. The DOM is fully updated by the time the watcher fires, and `autoFitSeats()`'s inner `setTimeout(0)` ensures measurements happen after paint.
- **Removed**: `autoFitSeats` call and `await fetchSeats()` from `selectArea()`
- **Added**: `watch(seats, () => { if (!editMode.value && seats.value.length > 0) autoFitSeats() })` — same pattern as the already-working `watch(editMode, ...)` path

## Late Session: Auto-Fit Bug Revelation (committed b89cf2f)

### The Real Bug: `!editMode.value` Guard on watch(seats)
- **Root cause**: The tree panel is `v-if="editMode"` — only rendered when editMode is true. Every time the user clicked an area in the tree, they were in edit mode. The guard `if (!editMode.value && ...)` evaluated to false every single time, silently blocking `autoFitSeats` from running on any area change.
- **The broken flow**: Click area (editMode=true) → seats change → watch fires → guard skips → no auto-fit. Then toggling to view mode hit `watch(editMode)` which correctly called `autoFitSeats` — explaining why one path "worked" and the other "didn't".

### Fix (b89cf2f)
1. **Removed the `!editMode.value` guard** from `watch(seats, ...)` — autoFitSeats now fires regardless of mode when seats change
2. **Added `{ flush: 'post' }`** to the seats watcher — ensures DOM is fully patched before canvas measurement
3. **Made `seatAreaTop` mode-aware** in `autoFitSeats()`: edit mode uses `paddedRowsAbove = 0` (compact grid, no padding), view mode uses `paddedRowsAbove = 100 - numRows` (100-row padded grid)
4. Changed `requestAnimationFrame` to `setTimeout(0)` inside autoFitSeats for simpler timing (rAF didn't help)
5. Removed unused `nextTick` import

### Key Technical Details
- `seatH = 40` matches actual CSS row height (32px seat + 8px margin-bottom via `--space-sm = 0.5rem`)
- Grid padding: view mode creates 100 grid rows (50 blank + 50 seat rows), edit mode creates 50 grid rows (all seat rows)
- `autoFitSeats` computes `rowsMap` from `seats.value` synchronously, then schedules canvas measurement with `setTimeout(0)`
- Seat area top offset in view mode: `(100 - numRows) * seatH` — shifts pan down by the padded rows' visual height
- In edit mode, fixed by logging the debug output
