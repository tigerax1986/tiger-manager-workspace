## Late‑Night Bug Fix – Runtime ReferenceError in TicketDesigner

### Error
- `Uncaught (in promise) ReferenceError: Cannot access 'selectedPlaceholder' before initialization at TicketDesigner.vue:571:33`
- Occurred when loading the TicketDesigner page after the QR‑code field improvements.

### Root Cause
- In Vue 3 `<script setup>`, reactive references (`ref`, `computed`) are evaluated in **top‑down order**.
- The `qrSizeMm` computed property (line ~553) referenced `selectedPlaceholder.value` before `selectedPlaceholder` was declared (line ~717).
- This created a **circular dependency** at runtime: `qrSizeMm` tried to read `selectedPlaceholder` while Vue was still initialising the script.

### Fix
1. **Moved the entire QR‑code block** (lines 553–603) **below** `const selectedPlaceholder = ref<CanvasPlaceholder | null>(null)`.
2. **Preserved all raw‑string bindings** (`qrSizeRaw`) and input/blur handlers.
3. **No changes to other numeric fields** (Width, Height, X, Y, etc.) – they already used raw buffers and were correctly ordered.

### Result
- Page loads without JavaScript errors.
- QR‑code size field behaves as intended: empty strings allowed, validation on blur, live sync when dragging.
- All other placeholder‑property inputs remain fully editable.

### Lesson
- **Declaration order matters** in Vue’s Composition API. Computed properties that depend on other refs must be defined **after** those refs.
- **Test after every structural change** – the bug only appeared after moving the QR‑code block earlier during the numeric‑input overhaul.
- **Runtime errors are immediate** – the page fails to load, making the issue obvious but blocking further testing.

**Status**: TicketDesigner is now error‑free and fully functional across all numeric inputs.
